Wi-Fi Service

This article provides information about the Hughes Wi-Fi Service.

About Hughes Wi-Fi Services

Hughes offers three varieties of WiFi Services via Aruba products:

  • WiFi 1.0 has Access Point (AP-93) at Site or location. Most sites have 1 AP-93. Some Sites have up to 3 AP-93 units (e.g., Bevmo).
  • Hospitality WiFi is used in scenarios where there are multiple access points and in many cases multiple floors (for instance Best Western)
  • WiFi 2.0 has multiple Instant Access Points aka IAP-105 or IAP-103 at site. Starting 11/17/14 sites with the IAP-105 device may be replaced with the IAP-103 (provides the same functionality).

Wi-Fi 1.0
  • Wireless connection to AP for internet connectivity.

Wi-Fi 2.0
  • Wireless connection to IAP for internet connectivity.
  • AP connects to PoE capable switch then to an Aruba virtual controller (on one of the IAPs.
  • Aruba Virtual controller provides configuration / software to the AP to for successful connection to internet.
Transport
  • Hughes or Customer-provided.
  • HR4700 (or Fortigate 60C) as the CPE/Router.
  • Optional Cisco 300 Switch for more than 4 APs.
Virtual Controller
  • Active IAP-105 may be designated as virtual controller for the other IAP-105 devices.
  • If IAP-105 goes down, next available IAP device will act as virtual controller by AWMS allowing for IAPs to run independently without connectivity to Hughes NOC.

**Transport for Wi-Fi Service must be either DSL/Cable/T1. If the site is a HAN and the secondary transport is EVDO or VSAT the Wi-Fi service will not work properly.

Rogue Services

Hughes Enterprise WiFi customers have the option of PCI Rogue Wireless Detection. Our services providescanning as well as optional Rogue Classification for devices that are attempting to access the customer's network.

  • Rogue Scanning only provides the identification of possible intrusions to the Wi-Fi network
    All Enterprise Wi-Fi networks are scanned for possible rogue devices intruding on the Wi-Fi network. An email notification is generated and sent to the network administrator detailing the MAC address and other information on the device in question.
  • Optional Rogue Classification allows for the customer to identify the device to the Wi-Fi network
    When the email notification of a possible intrusion, is received the network administrator will determine whether the device is rogue or a valid network device.

Troubleshooting via Network Management Portal (NMP)

   
  Check primary site's transport (line stats, speeds, etc.) and if issue is found troubleshoot accordingly
  Go to AP device details.
  Navigate to Actions section > Click Current Status to Ping the AP device. A Pop-up window with results will appear.
  • Is Responding = False indicates that the device is not responding in real time. Check physical connections. Go to Step 3.
  • Is Responding = True indicates that the device is responding in real time ping test. If AP's SNMP Uptime is less than 30 minutes, verify if an outage occurred (lost sync, etc.) recently. Note that AP recovery may take up to 45 minutes.
  Navigate to Performance Graphs section. Click on User Count to see the data for last 2 hours.
  If there are other Clients connected to the AP, the AP is functional. Troubleshoot the guest device.
  Click on User Count to see the data for last 2 hours. In the graph, identify if the usage has been constant. If not, there may be issues with the radio for the AP device.
  If AP is down, check if LAN MAC address is listed in the router.
  • In NMP and pull up the router device.
  • Go to Actions menu > Click Get Arp Cache to view ARP cache statistics.
  If AP's MAC address is listed in ARP Cache and AP shows down, verify if an outage occurred (lost sync, etc.) - recover may take up to 45 minutes.
  If AP's MAC address is not listed in the ARP Cache and physical connections were checked, create dispatch ticket.

Initial Troubleshooting Steps

   
  Verify that the user is in a location where there is Network Signal Strength showing on the device.
  Determine whether he or she connects regularly. If the user connects infrequently, troubleshoot the device settings first.
  Login to AWMS. Check that he or she is connected to the AP.
  Examine the AP association history for signs of ping-ponging.
  Look for changes in signal quality, using the Look for changes in signal quality, using the chart on the user details page.

Troubleshooting with a WiFi guest with their device

   
  Obtain the Store SSID that the user is attempting to connect to. Check with the user for the store Site ID or the store address.
  Note: Refer to FAQ section for commonly asked questions.

  • If no, ask the user to restart the browser and try again.
  If the issue is not resolved, check to see if there are any open cases on the Transport. Inform the caller about the existing issue. Troubleshoot Primary transport equipment at the Site.

Troubleshooting Guest not Seeing any WiFi service with Device

   
  Make sure that the WiFi is turned ON in your device.
  Look for a hardware switch or special function key on your laptop that can turn the wireless radio on or off -- it may have accidentally been turned off. Flip the switch or hit the function keys to test if this is the case. Also, sometimes the wireless radio may be disabled by power management settings on your laptop or your phone; fix this permanently by going to the Power Options for your device and changing the settings there so the wireless adapter won't be disabled by low battery triggers.
  If you're using a USB wireless network adapter, make sure it's plugged in correctly. On smart phones, you should also check whether wireless is enabled in the phone's main settings.

Slow WiFi or dropped connections

  1. Check the signal strength on the device used by the patron.
  2. If only 1-2 bars are showing, ask the patron to change his position closer to the IAP.

Wi-Fi troubleshooting for Android phones

   
  Power off - The first thing user should try is to power off the Android phone, remove the battery, leave the battery out for about 30 seconds, replace the battery, and power the phone back on.
  Power-cycle wireless. Sometimes, the phone retains a previous IP address from another wireless connection before and applies it to the current wireless. That's fine if the address scheme is the same and there is no address conflict, but that is not usually the case.

User can try turning off wireless and then turning it back on by following these steps:
  1. Tap the Menu button on your handset >Tap Settings >Tap Wireless and Network from within the Settings menu >Tap Wi-Fi settings from the Wireless and Network menu
  2. In the new screen, uncheck Wi-Fi to turn it off >After Wi-Fi is turned off, tap the entry again to turn it back on
  Forget the Network. If the previous method does not work, you might need to have your Android device forget the network and then re-add it. Your Android device cannot forget a wireless network that is within range, so you will need to get out of range of the network and then do the following:
  1. Long press the listed network you want to forget >Tap Forget Network.
  2. Get back within range and wait for the network to automatically show up.
  3. Your Android phone should have a new address and be working as you would expect.
  Wi-Fi Sleep Policy - If you are having persistent Wi-Fi issues, you can set the Wi-Fi sleep policy to Never and see if your issues stop. With this policy set to Never, it will drain your battery faster, but it will also keep your Wi-Fi connected on a more consistent basis.

SSID / Passphrase Changes

Process a SSID/Passphrase Change Ticket

  • Requesting information on a recent passphrase change.
  • Requesting a reset of the passphrase.

Verifying Customer Created a ENGGR case for SSID/Passphrase Change

   
  Log into CRM >Go to the VSAT Call Center Work List >Click on the Work Case URL.
  Verify in the customer's request that it contains the new values for the SSID/Passphrase and Swap Date in the Problem Description.
  Update the following fields:
  • Problem Type:Request SSID
  • Product:SSID
  • Case Type:Problem
  • Priority:High
  • Case Status:Open Escalated
  • Provider Group:Sustaining Network Engineering
  The primary engineer for the network should auto default, however if it doesn't, select the correct engineer from the dropdown. Information about the primary engineer can be obtained on the Employee Subscription tab.
  • Case will be closed by the assigned Network Engineer.
  • New SSID and Passphrase will be updated in Customer CRM by Network Engineer.
  • Customer will be notified of SSID/Passphrase change via Auto-Email Notification.
  • There will be a 48 hour overlap in the old and new authentication.

VerifyingCreation of a Remote case for the Retrieval of a Lost Passphrase

  Navigate to the company screen for that particular ticket by clicking on the search icon next to the Company ID on the Case page.
  Once the main information page is opened, navigate to the Notes tab > Notes Summary to locate the latest SSID data sheet attachment.
  • Add a note to the case with the SSID and Passphrase information.
  • Close the ticket.

*If the ticket is create via web. If the customer calls with passphrase request, follow standard operation procedures.

BKC Depot Maintenance Process for WiFi 1.0 Services

Creating a Special Projects Dispatch in CRM

   
  Identify that the site is a BKC Site and that it has Aruba WiFi Services.
  • BKC should be the first three letters for the SAN.
  • Site Type will indicate WIFI.
  Check in the Service Level tab to verify if the site has Depot Maintenance.Note: Follow standard dispatch procedures if "Depot Maintenance" is not listed under Service Level.
  Go to the Cust. Ticket # in the Customer Information section >Add "PJ4738::": If the Cust. Ticket # field is not yet populated.Note: This number MUST be included with two colons after the 8 to prevent WWTS from sending a technician to the site.
  Update all items listed in RED.Note: It is extremely important to have the Case Type: Special Project Dispatch. If this case type is not selected, the request will not be fulfilled correctly.
 
Use the following template in the Description field: "BKC advanced shipment requirement Project PJ4738: Process per SOW192. Site requires the shipment of an advanced replacement WiFi AP-93 device to be shipped UPS Over Night Next Day delivery from WWTS FRU stock in Louisville, KY. Shipment must be made same day as ticket issuance if received by WWTS by 3PM eastern time. Ship next business day if ticket received later. Ship quantity 1 of Hughes part number AE-1502992-0001. Contact WWTS Logistics with any issues (1-800-374-0002 Option 5, Option 1). Logistics must record part number and serial number of device shipped in ticket."
  Save the case >Click the Dispatch button.

Removing the AP from AWMS and the Controller - Actions Performed by a TL/MOD/ESC Control Desk

   
  Before the Special Project Dispatch is initiated, the TL/MOD/ESC Control Desk must disassociate the AP from the AWMS and the Controller. Validate that the site has Depot Maintenance in the CRM by reviewing the previous steps.
  Log into AWMS >Locate the site from the search field in the upper right corner of AWMS.
  Locate the correct AP > Select Modify Devices.
  Check the box of the specific AP device to be removed from the AWMS >Click the Delete button next to Delete selected devices:

Click the Apply Changes Now button.Note: A confirmation of the deletion will be listed in green font.
  Go to the Aruba Controller application by following the instruction provided by your supervisor. Select the Monitoring tab > Select All Access Points on the side menu of the Controller application.
Note: This will give add a search option at the top. Make sure that the AP Group is set to Any.
  Identify the correct AP by SAN in the Name column > Click the Clear button under the Actions column >Click the OK button on the confirmation pop-up screen.The AP will then be removed from the list.
Note: An error message may pop up in the Controller. Click the OK button - the application will still complete the removal process.
  Verify that the Special Project Dispatch was created properly in CRM after the AP has been removed from AWMS and the Controller.
